Chapter 01

AI와 함께 일하기 위한 기본 개념

마크다운 파일 (.md) 기초

MD 파일은 마크다운(Markdown) 형식으로 작성된 텍스트 파일입니다. 확장자가 .md 또는 .markdown으로 끝납니다.

📖 쉽게 말하면
일반 메모장 파일(.txt)과 비슷하지만, #을 앞에 붙이면 제목이 되고, **단어**로 감싸면 굵게 보이는 등 간단한 규칙으로 서식을 표현할 수 있는 파일입니다. 작성한 내용은 GitHub, Notion, Slack 등 다양한 플랫폼에서 자동으로 예쁘게 렌더링됩니다.

AI 도구들이 문서, 프롬프트, README, API 문서를 .md 형식으로 주고받기 때문에 자주 마주치게 됩니다.

# 이것이 제목입니다 (H1)
## 이것은 소제목 (H2)

일반 문장은 그냥 씁니다.

**굵게**  *기울임*  `코드`

- 목록 항목 1
- 목록 항목 2
💡 실무 팁
VS Code, Notion, Obsidian 등에서 .md 파일을 열면 서식이 적용된 미리보기를 볼 수 있습니다. Claude가 결과물을 .md로 줄 때, 그냥 Notion에 붙여넣으면 서식이 그대로 살아납니다.
LLM (대형 언어 모델) 핵심 개념

LLM (Large Language Model)은 ChatGPT, Claude, Gemini 등 AI 챗봇의 엔진입니다. 수천억 개의 텍스트 데이터를 학습해 사람처럼 글을 이해하고 생성하는 모델입니다.

📖 쉽게 말하면
인터넷에 있는 거의 모든 글을 읽고 암기한 독서왕과 대화하는 것과 비슷합니다. 다만 "학습 마감일(cutoff)"이 있어, 그 이후의 정보는 모릅니다.
모델 만든 곳 특징
GPT-4oOpenAI범용성 높음, 가장 널리 알려짐
Claude 3.5 SonnetAnthropic긴 문서 처리, 안전성 강조
Gemini 1.5 ProGoogleGoogle Workspace 연동
Llama 3Meta오픈소스, 자체 서버 운영 가능
토큰과 컨텍스트 윈도우 핵심 개념

토큰(Token)은 AI가 텍스트를 처리하는 최소 단위입니다. 영어 기준 약 4글자, 한국어는 약 1~2글자가 1토큰에 해당합니다. AI 서비스 비용은 사용한 토큰 수로 계산됩니다.

컨텍스트 윈도우(Context Window)는 AI가 한 번의 대화에서 기억할 수 있는 최대 토큰 수입니다. 이 한계를 넘으면 앞 내용을 잊습니다.

📖 쉽게 말하면
A4 용지 한 장을 쭉 읽어드릴 수 있는 메모리가 있다고 할 때, 그 범위 안에 있는 내용만 기억하는 것입니다. 긴 보고서를 통째로 주면 앞부분을 잊기 시작합니다.
💡 실무 팁
긴 문서를 AI에게 넘길 때는 핵심 섹션만 잘라서 주거나, "이 문서의 3페이지를 요약해줘" 식으로 범위를 좁혀주세요.
프롬프트 엔지니어링 핵심 기술

프롬프트 엔지니어링(Prompt Engineering)은 AI에게 원하는 결과를 끌어내기 위해 입력(프롬프트)을 전략적으로 설계하는 기술입니다.

같은 AI라도 어떻게 질문하느냐에 따라 결과물의 품질이 극단적으로 달라집니다. 프롬프트 엔지니어링은 AI를 잘 쓰는 핵심 역량입니다.

핵심 기법 5가지

1
역할 부여 (Role Prompting) — "너는 10년 경력의 마케터야. 아래 보고서를 리뷰해줘." 처럼 AI에게 전문가 페르소나를 부여합니다.
2
출력 형식 지정 — "JSON으로 출력해줘", "표로 정리해줘", "한 줄로 요약해줘" 처럼 원하는 형식을 명시합니다.
3
예시 제공 (Few-shot) — 원하는 결과물 예시를 1~3개 보여주면 AI가 그 패턴을 따라갑니다.
4
단계적 추론 유도 (Chain-of-Thought) — "단계별로 생각해줘", "먼저 ~를 분석하고, 그다음 ~를 해줘." 처럼 사고 과정을 단계로 나눕니다.
5
제약 조건 명시 — "500자 이내로", "~은 포함하지 말고", "경쟁사 이름은 언급하지 마." 처럼 금지 또는 범위 조건을 줍니다.
💡 황금률
역할 + 맥락 + 형식 + 제약을 모두 담은 프롬프트가 가장 좋은 결과를 냅니다. "요약해줘" 보다 "너는 편집장이야. 아래 보도자료를 300자 이내 뉴스레더 제목+소제목 형식으로 요약해줘."가 훨씬 강력합니다.
프롬프트 작성 vs 자연어 대화, 왜 이렇게 다른가? 개념

우리가 평소 카톡이나 회의에서 쓰는 자연어("그거 있잖아, 저번에 얘기한 거 한번 더 봐줘")는 상대가 맥락을 공유하고 있기 때문에 통합니다. AI는 그 맥락이 없습니다.

AI는 대화 기록 안에서만 맥락을 파악합니다. 새로운 대화창을 열면 이전 내용을 전혀 모릅니다. 따라서 프롬프트는 독립적인 업무 지시서처럼 써야 합니다.

구분 자연어 대화 프롬프트
맥락상대와 공유됨매번 직접 제공해야 함
생략가능 ("그거")불가 (구체적으로 명시)
형식비공식 OK구조화할수록 결과 좋음
수정"아 그게 아니라~"프롬프트 자체를 개선해야 함
📖 쉽게 말하면
AI에게 일을 시키는 건 새로운 아르바이트생에게 업무 지시서를 쓰는 것과 같습니다. 내가 당연하게 아는 회사 상황, 프로젝트 이름, 기대 형식을 모두 적어줘야 합니다. 익숙해지면 이 "업무 지시서(프롬프트)"를 저장해두고 재사용할 수 있습니다.
환각 현상 (Hallucination) 주의

AI가 사실이 아닌 정보를 마치 사실인 것처럼 자신있게 말하는 현상입니다. 존재하지 않는 논문, 틀린 수치, 없는 법률 조항을 그럴듯하게 생성합니다.

⚠️ 주의
AI 결과물의 수치, 인용, 법규, 고유명사는 반드시 원본 소스를 직접 확인하세요. "이 내용이 맞아?" 라고 AI에게 다시 물어봐도 또 틀린 내용을 확신하며 답할 수 있습니다.
💡 줄이는 방법
"확실하지 않으면 '모름'이라고 해줘", "인터넷을 검색해서 답해줘(검색 기능 있는 AI 사용 시)", "참고 소스도 함께 제공해줘" 같은 지시를 프롬프트에 포함하세요.
API와 API 명세서 핵심 개념

API (Application Programming Interface)는 서로 다른 소프트웨어가 정해진 규칙으로 데이터를 주고받는 통로입니다.

📖 쉽게 말하면
식당에서 메뉴판을 보고 "아메리카노 주세요"라고 하면 바리스타가 커피를 만들어줍니다. API는 이 메뉴판과 주문 방식의 규칙입니다. 손님(프로그램 A)은 규칙대로 주문하고, 주방(프로그램 B)은 규칙대로 응답합니다.

API 명세서는 이 메뉴판을 상세히 설명한 문서입니다. 어떤 주소(URL)로 요청을 보내고, 어떤 형식으로 데이터를 담고, 어떤 형태의 응답이 오는지를 기술합니다.

# API 명세서 예시 (간단화)
엔드포인트:  POST /api/send-message
요청 헤더:  Authorization: Bearer {API_KEY}
요청 본문:  {
              "to": "user@email.com",
              "message": "안녕하세요"
            }
응답:       {
              "status": "ok",
              "message_id": "msg_12345"
            }

REST API는 현재 가장 널리 쓰이는 API 방식입니다. HTTP 주소(URL)로 요청을 보내고 JSON 형식으로 데이터를 받습니다. Slack, Notion, Claude 등 거의 모든 SaaS 서비스가 REST API를 제공합니다.

💡 실무 포인트
AI 자동화를 구축할 때 "이 서비스 API 있어?"를 먼저 확인하세요. API가 있으면 사람이 클릭하지 않아도 프로그램이 자동으로 작업을 수행할 수 있습니다.
RAG — 검색 증강 생성 심화

RAG (Retrieval-Augmented Generation)는 AI가 답변하기 전에 우리 회사 문서나 데이터베이스를 검색해서 근거를 찾아 답하는 방식입니다.

📖 쉽게 말하면
시험을 볼 때 오픈북(교재 참고 가능)으로 치르는 것과 같습니다. AI가 기억만으로 답하는 게 아니라, 우리 사내 문서를 검색하고 그 내용을 근거로 답합니다. 환각 현상을 줄이고 최신 사내 정보를 반영할 수 있습니다.
사용자 질문
→
사내 문서 검색
→
관련 청크 추출
→
LLM 답변 생성
→
결과 + 출처

Notion AI, Microsoft Copilot for 365, 사내 GPT 구축 등이 RAG를 활용하는 대표 사례입니다.

시스템 프롬프트 vs 사용자 프롬프트 개념

AI에게 전달되는 지시는 크게 두 층위가 있습니다.

구분 시스템 프롬프트 사용자 프롬프트
작성자서비스 개발자 / 관리자일반 사용자
위치대화창 뒤에 숨겨져 있음사용자가 입력하는 채팅창
역할AI의 성격·규칙·역할을 설정실제 업무 요청
예시"너는 부스터스 AX 도우미야. 항상 한국어로 답해.""이번 주 교육 자료 요약해줘"
💡 실무 포인트
우리 팀용 AI 도구를 만들 때, 시스템 프롬프트에 회사 가이드라인·말투·역할을 박아두면 누가 써도 일관된 결과물이 나옵니다.

로컬 환경 vs 웹 환경 기초

로컬(Local)은 내 컴퓨터에서만 작동하는 환경입니다. 파일을 만들고 브라우저에서 열어도 내 PC에서만 보입니다. 웹(Web)은 인터넷에 연결된 서버에 올려 누구나 URL로 접근할 수 있는 환경입니다.

구분 로컬 웹 (배포 후)
접근 가능 범위내 컴퓨터만인터넷이 있으면 누구나
URLfile:///Users/.../index.htmlhttps://my-app.vercel.app
서버 필요 여부불필요필요 (Vercel 같은 호스팅 서비스)
비용없음소규모는 무료 서비스 사용 가능
💡 자주 하는 실수
AI가 만들어준 HTML 파일을 내 컴퓨터에서 열면 잘 보이는데, 카톡으로 링크를 보내면 안 열립니다. 로컬 파일은 링크 공유가 불가능합니다. 공유하려면 반드시 배포가 필요합니다.
프론트엔드 vs 백엔드 기초

웹 서비스는 크게 두 영역으로 나뉩니다.

구분 프론트엔드 (Frontend) 백엔드 (Backend)
위치사용자 화면(브라우저)서버(보이지 않는 곳)
역할UI 렌더링, 클릭 처리데이터 저장, 로직 처리, API
언어HTML, CSS, JavaScriptPython, Node.js, Java, Go 등
비유식당 홀 (손님이 보는 곳)주방 (손님이 못 보는 곳)
📖 쉽게 말하면
Notion 앱을 열었을 때 보이는 화면, 버튼, 레이아웃은 프론트엔드입니다. 내가 내용을 저장하면 실제로 Notion 서버에 데이터가 저장되는 그 과정은 백엔드입니다.

AI가 만들어주는 HTML 파일은 대부분 프론트엔드 전용입니다. 데이터를 저장하거나 로그인 기능이 필요하면 백엔드가 필요합니다.

HTML · CSS · JavaScript 기초 기초
언어 역할 비유
HTML 뼈대 — 제목, 문단, 버튼, 이미지 등 구조 정의 건물의 골조
CSS 스타일 — 색상, 폰트, 간격, 애니메이션 등 시각 처리 인테리어 · 외장
JavaScript 동작 — 클릭 이벤트, 데이터 로드, 상호작용 처리 전기 · 설비 (움직이게 만드는 것)
💡 실무 포인트
AI에게 "이 HTML 파일에서 배경색을 바꿔줘"처럼 요청하면 됩니다. 세 가지를 직접 외울 필요는 없지만, 어떤 역할인지 알면 AI에게 더 정확한 수정 요청을 할 수 있습니다.
GitHub — 코드 저장 & 버전 관리 툴 도구

GitHub는 코드 파일을 클라우드에 저장하고, 변경 이력을 관리하는 플랫폼입니다. 개발자들의 구글 드라이브라고 생각하면 됩니다. 단, 파일 변경 내역(누가 언제 무엇을 바꿨는지)을 완벽하게 추적합니다.

📖 쉽게 말하면
파일명에 _최종.docx → _최최종.docx → _진짜최종.docx를 붙이는 대신, GitHub가 자동으로 모든 버전을 저장하고 이전 버전으로 되돌릴 수 있게 해줍니다.

핵심 용어

용어의미
Repository (레포)하나의 프로젝트 폴더. 코드와 파일이 담기는 공간

Push내 컴퓨터의 변경사항을 GitHub 서버에 업로드

Branch독립적으로 작업하는 분기. 실험적 변경사항을 별도로 관리

코드를 수정할 때마다 Commit → Push
Vercel — 무료 배포 플랫폼 도구

Vercel은 GitHub에 올려둔 코드를 인터넷에 즉시 배포해주는 플랫폼입니다. 클릭 몇 번으로 https://프로젝트명.vercel.app 같은 URL이 생깁니다. 개인/소규모 프로젝트는 무료입니다.

📖 쉽게 말하면
GitHub가 내 코드를 보관하는 창고라면, Vercel은 그 창고에서 물건(코드)을 꺼내 진열장(웹사이트)에 올려놓는 역할입니다.

Vercel 배포 과정

1
vercel.com에서 GitHub 계정으로 로그인
2
Add New Project → GitHub 레포지토리 선택
3
Deploy 클릭 → 약 1분 후 URL 자동 생성
4
이후 GitHub에 코드를 Push할 때마다 자동으로 재배포
💡 실무 포인트
Vercel은 GitHub와 연결하면 코드가 바뀔 때마다 자동으로 최신 버전을 배포합니다. "수정하고 저장하면 바로 반영된다"고 이해하면 됩니다. 부스터스 AX 대시보드(boosters-ax.vercel.app)가 이 방식으로 운영됩니다.
Google 베이스 로그인 (OAuth) 만들기 도구

내가 만든 웹 서비스에 "Google 계정으로 로그인" 버튼을 붙이는 방법입니다. OAuth 2.0이라는 표준 방식을 사용하며, 사용자의 비밀번호를 직접 다루지 않아 안전합니다.

📖 쉽게 말하면
특정 클럽(내 서비스)에 입장할 때 신분증 대신 "Google이 이 사람 구글 계정 맞아요"라는 확인서를 받아 입장시켜 주는 방식입니다. 비밀번호는 Google이 관리하고, 우리 서비스는 "신원 확인됨" 여부만 받습니다.

구현 절차 (개요)

1
Google Cloud Console에서 프로젝트 생성 → OAuth 동의 화면 설정
2
클라이언트 ID / 클라이언트 시크릿 발급 (API 키와 비슷한 개념)
3
서비스 코드에 Google 로그인 라이브러리(next-auth, Firebase Auth 등) 연결
4
로그인 버튼 클릭 시 Google 로그인 화면으로 이동 → 완료 후 내 서비스로 복귀
💡 실무 포인트
도메인을 회사 것(@boosters.kr 등)으로 제한하면 외부인의 접근을 막을 수 있습니다. 내부 전용 대시보드에 유용합니다.
도메인과 DNS 개념

도메인(Domain)은 인터넷 주소 이름입니다. google.com, boosters-ax.vercel.app이 도메인입니다. 실제로 서버는 142.250.80.46 같은 숫자(IP 주소)로 구분되지만, 사람이 기억하기 어려워 이름으로 매핑합니다.

DNS (Domain Name System)는 도메인 이름을 실제 서버 IP 주소로 변환해주는 전화번호부 역할을 합니다.

boosters-ax.vercel.app 입력
→
DNS 조회
→
76.76.21.21 (서버 IP)
→
페이지 로드
💡 실무 포인트
Vercel이 제공하는 .vercel.app 도메인은 무료입니다. 회사 브랜드 도메인(예: ax.boosters.co.kr)을 쓰려면 도메인을 구입하고 Vercel 설정에서 연결하면 됩니다. 도메인 구입은 가비아, Cloudflare 등에서 가능합니다.
환경변수 (Environment Variables) 개념

환경변수는 코드 밖에서 관리하는 비밀 설정값입니다. API 키, 비밀번호, DB 주소처럼 코드에 직접 쓰면 안 되는 민감한 정보를 별도 파일이나 서비스 설정에 저장합니다.

📖 쉽게 말하면
레시피(코드) 안에 "소금 200g"이라고 쓰는 대신, "소금은 냉장고 환경변수 파일에 있어"라고 쓰는 것입니다. 레시피가 외부에 공개돼도 소금의 실제 양(API 키 값)은 노출되지 않습니다. 
# .env 파일 (절대 GitHub에 올리면 안 됨)
GOOGLE_CLIENT_ID=123456789-abc.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=GOCSPX-xxxxxxxxxxxx
DATABASE_URL=postgresql://user:pass@host/db
⚠️ 주의
.env 파일은 절대 GitHub에 올리지 마세요. API 키가 노출되면 악용될 수 있습니다. Vercel에는 대시보드의 Environment Variables 메뉴에서 별도로 입력합니다.

자동화 프로세스를 만들고 싶다면

어떤 업무를 자동화해야 하는가 전략

모든 업무를 자동화할 수 있는 건 아닙니다. 자동화 ROI가 높은 업무는 반복적이고 규칙이 명확하며 수치화 가능한 것들입니다.

자동화 적합 자동화 부적합
주간 리포트 수집 및 취합 고객과 신뢰 쌓는 미팅
신규 입사자 온보딩 이메일 발송 복잡한 이해관계자 조율
KPI 수치 자동 집계 창의적 전략 기획
SNS 예약 게시 위기 상황 커뮤니케이션
계약서 초안 생성 최종 계약 협상

자동화 대상 발굴 체크리스트

✓
이 작업을 한 달에 몇 번 하는가? 4회 이상이면 후보
✓
할 때마다 같은 순서로 진행하는가? 예스면 자동화 가능
✓
결과물을 보고 맞다/틀리다를 판단할 기준이 있는가? 있으면 검증 자동화도 가능
✓
이 작업에 쓰는 시간이 월 2시간 이상인가? 그렇다면 자동화 투자 가치 있음
확인해야 하는 문서와 파일 정의 전략

자동화를 구축하기 전에 어떤 데이터를 어디서 가져오고, 어디에 쓰는지를 명확히 정의해야 합니다.

1
입력 소스 정의 — 어떤 파일/서비스에서 데이터가 시작되는가?
(예: Google Sheets, Notion DB, 이메일 수신함, Slack 채널)
2
트리거 정의 — 언제 자동화가 시작되는가?
(예: 특정 시간, 파일 변경 시, 폼 제출 시, 키워드 수신 시)
3
처리 로직 정의 — 데이터를 어떻게 가공하는가?
(예: 요약, 번역, 분류, 계산, 필터링)
4
출력 형식 정의 — 결과물이 어떤 형태여야 하는가?
(예: Slack 메시지, 이메일, Notion 페이지, 엑셀 파일)
💡 실무 팁
이 4가지를 A4 한 장에 그림으로 그려보세요. "데이터가 어디서 와서 어디로 가는지" 흐름이 보이면 자동화 설계의 절반은 끝납니다.
게시 채널 선택 전략

자동화 결과물을 어디에 보낼지는 수신자의 행동 패턴에 맞게 선택합니다.

채널 특징 적합한 케이스
Slack / Teams 즉시성 높음, 부담 낮음 빠른 알림, 일일 리포트, 이상 감지 경보
이메일 공식성 높음, 기록 보존 주간 요약, 외부 발송, 계약·승인 관련
Notion / Confluence 검색 가능, 협업 문서화 누적 기록, 레퍼런스 문서, Wiki
Google Sheets 수치 관리, 필터·정렬 가능 KPI 대시보드, 데이터 집계
웹 대시보드 시각화, 권한 관리 가능 임원 보고, 전사 공개 지표
💡 실무 포인트
"중요한 알림은 Slack, 공식 기록은 Notion, 수치는 Sheets" 처럼 채널별 역할을 팀 내에서 합의해두면 자동화 결과물이 어디에 있는지 헷갈리지 않습니다.
웹훅 (Webhook) 기술 개념

웹훅은 특정 이벤트가 발생했을 때 지정한 URL로 자동으로 데이터를 보내주는 방식입니다. "무언가가 일어났을 때 나에게 알려줘"를 자동화하는 핵심 메커니즘입니다.

📖 쉽게 말하면
"손님이 문을 열면 종이 울리는" 가게 문 벨과 같습니다. 내가 계속 확인하지 않아도, 이벤트(손님 입장)가 생기면 자동으로 신호(HTTP 요청)를 보냅니다.
이벤트 발생
(폼 제출, 결제 완료 등)
→
Webhook URL로
데이터 전송
→
받는 쪽에서
자동 처리

Slack, Notion, GitHub, Stripe, Google Forms 등 대부분의 SaaS 서비스가 웹훅을 지원합니다. 노코드 자동화 툴(n8n, Zapier)에서 이 웹훅을 시작점으로 사용합니다.

스케줄러 / 크론잡 (Cron Job) 기술 개념

스케줄러는 특정 시간이나 주기에 자동으로 작업을 실행하는 장치입니다. 크론잡(Cron Job)은 서버에서 이를 구현하는 전통적인 방법으로, 시간을 5자리 코드로 표현합니다.

# 크론 표현식: 분 시 일 월 요일
0 9 * * 1-5   # 평일 매일 오전 9시
0 18 * * 5    # 매주 금요일 오후 6시
0 0 1 * *     # 매월 1일 자정
*/30 * * * *  # 30분마다
💡 실무 포인트
직접 코드를 짜지 않아도 됩니다. n8n, Zapier, Make.com 같은 노코드 툴에서 "매일 오전 9시에 실행"을 클릭 몇 번으로 설정할 수 있습니다. 크론 표현식은 AI에게 "매주 월요일 오전 10시를 크론 표현식으로 바꿔줘"라고 하면 즉시 알려줍니다.
노코드 자동화 툴 도구

코딩 없이 드래그&드롭으로 자동화 흐름을 만드는 도구들입니다. "어떤 앱에서 어떤 일이 생기면, 다른 앱에서 어떻게 행동해"를 시각적으로 연결합니다.

툴 특징 가격
Zapier 가장 많은 앱 연동(7,000+), 쉬운 UX 무료 플랜 있음 (월 100회)
Make (구 Integromat) 복잡한 분기 처리 가능, 시각적 플로우차트 무료 플랜 있음 (월 1,000 ops)
n8n 오픈소스, 자체 서버 설치 가능, 무제한 셀프호스팅 시 무료
Claude / GPT + API AI 처리(요약, 분류, 생성) 포함 자동화 토큰 과금
